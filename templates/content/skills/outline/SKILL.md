# Outline Skill

## Purpose

Create well-structured outlines that guide content creation and ensure logical flow and complete coverage.

## When to Use

- After research is complete
- Before writing begins
- When restructuring existing content
- When content feels disorganized
- For collaborative content planning

## Outline Levels

### High-Level Outline

**Purpose:** Overall structure and flow

**Format:**
```
I. Introduction
II. Main Topic 1
III. Main Topic 2
IV. Main Topic 3
V. Conclusion
```

**Use Case:** Initial planning, stakeholder approval

### Detailed Outline

**Purpose:** Section-by-section guidance

**Format:**
```
I. Introduction
   A. Hook
   B. Context
   C. Thesis

II. Main Topic 1
   A. Subtopic 1
      1. Detail
      2. Example
   B. Subtopic 2
```

**Use Case:** Writer guidance, content planning

### Writing Outline

**Purpose:** Paragraph-level planning

**Format:**
```
## Introduction

### Opening Hook
[Specific approach]

### Context
[Background to provide]

### Thesis Statement
[Exact or approximate wording]

## Section 1

### Topic Sentence
[Main point of paragraph]

### Supporting Points
- Point 1
- Point 2

### Example
[Specific example to use]
```

**Use Case:** Detailed writing guide

## Outline Creation Process

### 1. Identify Key Messages

**From Research:**
- List all key findings
- Group related points
- Identify main themes
- Prioritize by importance

**Audience Needs:**
- What must they know?
- What should they know?
- What would be nice to know?

### 2. Determine Structure

**Common Patterns:**

**Problem-Solution:**
```
1. Define the problem
2. Explain impact
3. Present solution
4. Show implementation
5. Discuss results
```

**Chronological:**
```
1. Background/history
2. Current state
3. Recent developments
4. Future outlook
```

**Hierarchical:**
```
1. Overview
2. Component 1
3. Component 2
4. Component 3
5. Integration
```

**Compare-Contrast:**
```
1. Introduction
2. Option A details
3. Option B details
4. Comparison
5. Recommendation
```

### 3. Arrange Logically

**Flow Principles:**
- Simple to complex
- Known to unknown
- General to specific
- Problem to solution

**Transitions:**
- Each section leads to next
- No gaps in logic
- Build on previous concepts

### 4. Add Details

**For Each Section:**
- Key points to cover
- Examples to include
- Visuals needed
- Word count estimate

## Outline Templates

### Article Outline

```markdown
# [Title]

## Introduction (100-150 words)
- Hook: [approach]
- Context: [background]
- Thesis: [main point]

## Section 1: [Title] (300-400 words)
- Main point
- Supporting details
- Example

## Section 2: [Title] (300-400 words)
- Main point
- Supporting details
- Example

## Section 3: [Title] (300-400 words)
- Main point
- Supporting details
- Example

## Conclusion (100-150 words)
- Summary
- Call to action
- Next steps

## Estimated Total: 1200-1500 words
```

### Tutorial Outline

```markdown
# Tutorial: [Task]

## Prerequisites
- [Requirement 1]
- [Requirement 2]

## Learning Objectives
- [Objective 1]
- [Objective 2]

## Step 1: [Action] (200-300 words)
- Explanation
- Code example
- Expected result

## Step 2: [Action] (200-300 words)
- Explanation
- Code example
- Expected result

## Step 3: [Action] (200-300 words)
- Explanation
- Code example
- Expected result

## Complete Example
- Full working code

## Troubleshooting
- Common issue 1
- Common issue 2

## Next Steps
- Where to go from here
```

```markdown
- Type

- Version history
```

## Outline Review

### Self-Review Checklist

**Structure:**
- [ ] Logical flow
- [ ] No gaps
- [ ] No redundancy
- [ ] Appropriate depth

**Coverage:**
- [ ] All key points included
- [ ] Examples planned
- [ ] Edge cases noted
- [ ] Prerequisites stated

**Audience:**
- [ ] Appropriate level
- [ ] Clear progression
- [ ] Engaging structure
- [ ] Actionable content

### Peer Review

**Questions for Reviewers:**
- Does the flow make sense?
- Are any topics missing?
- Is any section too detailed/not detailed enough?
- Would this structure help you write?

## Outline Best Practices

### Do's

- Start with key messages
- Use parallel structure
- Include examples in outline
- Estimate word counts
- Note visual requirements
- Plan transitions

### Don'ts

- Don't over-detail (leave room for writing)
- Don't lock into structure (be flexible)
- Don't skip review step
- Don't forget conclusion

## Revision Process

### When to Revise

- Research reveals new information
- Writing reveals structural issues
- Feedback suggests changes
- Scope changes

### How to Revise

1. Identify the issue
2. Consider options
3. Update outline
4. Communicate changes
5. Continue writing
